### Detected Architectural Risks

**RISK-A (HIGH): Dual execution paths not unified**
`/api/v2/task` → `MissionSystem` → `JarvisOrchestrator` (background task)
`/run` → `MetaOrchestrator.run()` → `JarvisOrchestrator`
These two paths run the same orchestrator but through different wrappers, storing state in two different objects (`MissionSystem` record vs `MissionContext`). A mission submitted via `/api/v2/task` is NOT tracked by `MetaOrchestrator._missions`, and a mission run via `/run` is NOT in `MissionSystem`. This creates an irreconcilable split in observability.

**RISK-B (MEDIUM): `_check_auth` defined in 3 places**
- `api/_deps.py` (canonical)
- `api/main.py` (local redefinition, line 254)
- `api/routes/monitoring.py` (different signature — only 1 argument)

The `monitoring.py` variant only validates `token`, not `authorization: Bearer`, making monitoring endpoints weaker than the rest of the API.

**RISK-C (MEDIUM): `OrchestratorV2` unreachable from API**
`use_budget=False` is hardcoded in `MetaOrchestrator.run_mission()` and there is no API parameter to set `use_budget=True`. The V2 orchestrator with budget/DAG capabilities is functionally dead code from the API perspective.

**RISK-D (LOW): Dual trace system**
`core/trace.py` (MissionTrace) and `core/orchestration/decision_trace.py` (DecisionTrace) serve overlapping purposes. Only DecisionTrace is used in MetaOrchestrator. MissionTrace appears to be a legacy system with no active callers in the main execution path.

**RISK-E (LOW): `api/main.py` `_check_auth` re-import shadow**
Line 249 imports `_check_auth` from `api._deps`, then line 254 redefines it locally. The local definition is identical but creates confusion and a maintenance trap.

## PHASE 2 — ORCHESTRATION ANALYSIS

### Findings

**State Machine (MetaOrchestrator)**
The state machine is deterministic and well-validated. Transitions are explicitly constrained via `_VALID_TRANSITIONS`. Invalid transitions raise `ValueError` and are caught. The REVIEW state correctly allows re-entry to RUNNING.

**Task Decomposition**
`MissionClassifier` → `ContextAssembler` → `PreExecutionAssessment` → `ExecutionSupervisor` is a clean pipeline. Each phase is wrapped in `try/except` with fail-open semantics, meaning classification or context failures degrade gracefully rather than crashing.

**Retry Logic**
`ExecutionSupervisor` has `_MAX_RETRIES = 2` with `1.5s` exponential backoff. This is conservative and appropriate for a v1 system.

**Fallback Logic**
JarvisOrchestrator's `INTENT_MAP` has `"default": "shadow-advisor"` — a guaranteed local fallback that requires no external API. Good.

**Weak Points**

1. **Context explosion risk**: `ContextAssembler` enriches the goal with prior skills + memories before passing to the LLM. There is no hard token budget guard on `planning_prompt_context()`. A mission with many prior failures or skills could produce an excessively large prompt.

2. **`run()` backward-compat branch**: The `run()` shim in MetaOrchestrator has a conditional: `if mode == "auto" and not session_id` → uses `run_mission()`, otherwise falls back to `self.jarvis.run()`. This means non-"auto" mode calls bypass the full MetaOrchestrator pipeline (classification, reflection, learning loop, memory storage). This is a significant gap.

3. **No concurrency guard on `_missions` dict**: `MetaOrchestrator._missions` is a plain dict mutated from async coroutines. Under uvicorn with multiple concurrent requests, this is safe only because Python's GIL protects simple dict operations, but it is fragile and will break under `anyio` worker threads or multiprocessing.

4. **`awaiting_approval` state does not transition**: When `outcome.error_class == "awaiting_approval"`, the mission stays in `RUNNING` state. There is no timeout or cleanup if the approval never comes. The mission leaks in `_missions`.

**Orchestration Reliability Rating: 7 / 10**

## PHASE 4 — EXECUTION LAYER VALIDATION

### Execution Safety Architecture

The `ExecutionPolicy` is deterministic, pure-Python, and well-defined:
- `_CRITICAL_ACTIONS` always require approval: `self_modify`, `modify_config`, `install`, `network`, `restart_service`
- `_SAFE_ACTIONS` allowed without approval: `read`, `write`, `execute`
- Risk thresholds are explicit per action type

`ExecutionEngine` (executor/execution_engine.py) is a proper threaded queue with:
- Priority heap
- Explicit status transitions (PENDING → RUNNING → SUCCEEDED/FAILED/TIMED_OUT/CANCELLED)
- No silent task drops
- 4 concurrent workers max

### Issues

**EXEC-1 (MEDIUM): Reasoning layer can bypass execution layer**
`missions.py:submit_task()` runs a background async closure (`_run_mission`) that calls `_get_orchestrator().run()` directly. This path skips `ExecutionEngine` entirely. The ExecutionEngine is a separate system that appears to run independently from the main mission pipeline. It is unclear what tasks go through `ExecutionEngine` vs the orchestrator's direct invocation.

**EXEC-2 (LOW): `dry_run` propagation**
`DRY_RUN=true` is set in settings but there is no evidence that `dry_run` is checked inside `ExecutionPolicy` or `ActionExecutor` before executing external calls. If it is, the check is not visible at the policy layer.

**EXEC-3 (LOW): `action_executor` daemon**
`core/action_executor.py` is started as a daemon thread at `main.py` startup. Its relationship to `ExecutionEngine` is unclear — are these the same action queue or two independent ones?

**Execution Safety Score: 7 / 10**

## PHASE 5 — MEMORY AND GROUNDING

### Memory Architecture

The `MemoryFacade` is a clean abstraction over multiple backends. Its fail-open design (each backend failure is caught independently) is appropriate for a system where memory is enhancement, not requirement.

The context assembly pipeline in `ContextAssembler` is structured:
1. Retrieve prior skills (relevant to goal)
2. Retrieve relevant memories
3. Inject as `planning_prompt_context()` into enriched goal

### Issues

**MEM-1 (MEDIUM): No context budget enforcement**
`planning_prompt_context()` has no token limit. Skills and memories concatenated without a hard cap could produce prompts exceeding model context windows, causing silent truncation or errors.

**MEM-2 (MEDIUM): Memory stored on both success AND failure paths, but in different formats**
Success: `store_outcome(content=f"Mission {mid}: {goal} -> {result}")`
Failure: `store_failure(content=f"Mission {mid} FAILED: {goal} -> {error}")`
Both are plain text strings. There is no structured memory record with goal type, duration, agent used, etc. Retrieval quality will degrade as corpus grows.

**MEM-3 (LOW): Dual vector stores**
`memory/vector_store.py` and `memory/vector_memory.py` appear to be separate implementations. It is unclear which one is active and whether both are written to.

**Memory Efficiency Score: 6.5 / 10**
**Grounding Reliability Score: 7 / 10**

## PHASE 6 — TRACE AND EVENT SYSTEM

### Trace System Analysis

`DecisionTrace` (used in MetaOrchestrator) is the active, canonical trace:
- Records: classify → retrieve → plan → pre_check → execute → reflect → learn → store → complete
- Saves to `workspace/traces/<mission_id>.jsonl`
- Includes cost tracking (`record_cost`) and human summary

`MissionTrace` (core/trace.py) also writes to `workspace/missions/<id>/trace.jsonl` — appears to be a legacy system not called from the main pipeline.

`JarvisSession.event_stream` (core/events.py) handles real-time event streaming for WebSocket subscribers.

### Issues

**TRACE-1 (HIGH): trace_id is actually mission_id**
There is no distinct `trace_id` concept. The "trace" IS the `mission_id`. The GET `/api/v1/trace/{trace_id}` endpoint referenced in the audit brief — this route does not appear in the API routes scanned. If it exists, it is not in the main route modules.

**TRACE-2 (MEDIUM): Dual trace files per mission**
A single mission execution writes two separate trace files at different paths. Consumers (dashboard, monitoring) must know which file to read. No unified trace reader exists.

**TRACE-3 (MEDIUM): Fail-open on all critical phases**
Every phase in `run_mission()` (classify, assemble, pre-assess, reflect, learn, skill-record, memory-store) is wrapped in `try/except: pass`. While this ensures forward progress, it means classification failures, reflection failures, and memory storage failures are all silently swallowed. Only the structlog warning at the exception level distinguishes a degraded run from a healthy one — and these inner exceptions don't even log a warning.

**Trace Integrity Score: 6 / 10**
**Event Lifecycle Reliability Score: 7 / 10**

## PHASE 8 — SECURITY REVIEW

### Findings

**Auth mechanism:**
- `JARVIS_API_TOKEN`: simple token comparison (X-Jarvis-Token or Authorization: Bearer)
- `JARVIS_SECRET_KEY`: used as JWT signing secret AND admin password
- JWT issued via `POST /auth/token` (admin / JARVIS_SECRET_KEY)
- Fall-through: if `JARVIS_API_TOKEN` is empty (`""`), ALL endpoints are unauthenticated

**Token validation:**
- Issued JWTs have `exp` field — expiry enforced at creation
- BUT: `_check_auth()` in `_deps.py` does NOT verify JWT signature or expiry — it does a raw string comparison of the provided token against `JARVIS_API_TOKEN`. A valid JWT issued by `/auth/token` will NOT pass `_check_auth()` unless `JARVIS_API_TOKEN` happens to equal the JWT string. This is a broken auth architecture: two auth systems that don't interoperate.

**Default secret:**
`JARVIS_SECRET_KEY` defaults to `"change-me-in-production"`. If not overridden in environment, JWT tokens are signed with a known key.

**Sensitive data:**
- `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY` loaded from environment — no evidence of accidental logging of these values in structlog calls.
- APK files (`.apk`) committed directly to the repository root — not a security risk per se but increases repo size and creates unnecessary exposure of compiled binaries.

**Security Posture Score: 5.5 / 10**

Critical finding: The JWT auth system (`/auth/token` → JWT) and the token auth system (`_check_auth` → raw string) are disconnected. Issuing a JWT token grants no actual API access unless `JARVIS_API_TOKEN` is set to the same JWT string, which is not how JWT works.

## STRENGTHS

1. **MetaOrchestrator state machine**: Clean, deterministic, logged, validated. This is the architectural backbone and it is solid.

2. **`core/state.py` as single source of truth**: `MissionStatus`, `RiskLevel`, `SessionStatus` all live in one place. No duplication in the main branch.

3. **`config/settings.py`**: All configuration in one place, environment-driven, typed with sensible defaults.

4. **`ExecutionPolicy`**: Deterministic, pure-Python, no external dependencies. Risk classification is explicit and auditable.

5. **`ExecutionEngine`**: Proper threaded task queue with priority, retry, and status tracking. No silent drops.

6. **`MemoryFacade`**: Clean abstraction over multiple backends. Fail-open design prevents memory failures from breaking missions.

7. **`DecisionTrace`**: Structured, append-only, per-mission audit log. Enables full explainability.

8. **Fail-open orchestration pipeline**: Classification, context assembly, and reflection failures degrade gracefully rather than aborting the mission.

9. **`api/_deps.py`**: Shared auth and dependency injection in one file. Good pattern.

10. **Test coverage**: 262+ tests covering key subsystems. Architecture coherence tests (`test_architecture_coherence.py`, `test_beta_architecture.py`) actively enforce single-source-of-truth invariants.

## WEAKNESSES

1. **Dual execution paths**: `/api/v2/task` and `/run` use different pipelines with different state tracking. This is the #1 architectural weakness.

2. **JWT issued but never verified on incoming requests**: `/auth/token` issues proper JWTs, but `_check_auth()` does raw string comparison. JWT tokens are functionally useless for API access.

3. **Backward-compat `run()` shim bypasses MetaOrchestrator pipeline** for non-"auto" modes.

4. **`_check_auth` defined in 3 places** with different signatures — monitoring endpoints have weaker auth.

5. **Silent exception swallowing in all MetaOrchestrator phases**: `try/except: pass` on classify, assemble, reflect, learn, store — failures are invisible in logs.

6. **No context budget guard**: Memory + skill injection into prompts has no token limit.

7. **`awaiting_approval` state leaks**: Missions stuck in approval never time out and are never cleaned from `_missions`.

8. **Two trace systems writing to different paths**: Operational confusion, no unified reader.

## RECOMMENDED ACTIONS

### P0 — Security (do before any public exposure)

**P0-1: Fix JWT auth**
`_check_auth()` must validate incoming JWT tokens via `jwt.decode()`, not raw string comparison.
Alternatively, deprecate JWT issuance and use only the static `JARVIS_API_TOKEN` model.
Do not run both systems disconnected.

**P0-2: Enforce non-default secret**
Add a startup validation: if `JARVIS_SECRET_KEY == "change-me-in-production"`, log a `CRITICAL` warning and optionally refuse to start in production mode.

### P1 — Architectural (do before v2 expansion)

**P1-1: Unify execution paths**
`POST /api/v2/task` should call `MetaOrchestrator.run_mission()` directly (or via a thin adapter), not `MissionSystem + JarvisOrchestrator` in a background task. This makes all missions visible in one state store.

**P1-2: Deduplicate `_check_auth`**
Remove the redefinition in `api/main.py` (line 254) and fix `api/routes/monitoring.py` to use the canonical 2-argument signature from `api/_deps.py`.

**P1-3: Add context budget to ContextAssembler**
`planning_prompt_context()` should enforce a hard character/token limit (e.g. 4000 chars) before injecting into the enriched goal.

**P1-4: Add approval timeout**
Missions in `awaiting_approval` state for more than N minutes (configurable, default 30min) should auto-transition to `FAILED` with reason `approval_timeout`.

### P2 — Reliability

**P2-1: Log inner exceptions in MetaOrchestrator pipeline**
Replace bare `except: pass` in classify/assemble/reflect/learn/store phases with `except Exception as e: log.warning("phase_failed", phase="classify", err=str(e)[:80])`.

**P2-2: Deprecate `MissionTrace` (core/trace.py)**
`DecisionTrace` covers the same ground with more structure. Remove or archive `MissionTrace` to eliminate the dual trace system.

**P2-3: Fix `run()` shim to route all modes through `run_mission()`**
The mode-based fallback to `self.jarvis.run()` in `MetaOrchestrator.run()` bypasses classification, reflection, and learning. All modes should flow through the full pipeline.

**P2-4: Make `_missions` dict thread-safe**
Wrap `MetaOrchestrator._missions` accesses in an `asyncio.Lock()` for concurrent mission safety.

### P3 — Repository

**P3-1: Remove APK files from git**
